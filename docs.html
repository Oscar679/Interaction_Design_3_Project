Architecture and implementation details of the Campus Weather application

Abstract Service Class

The base class for all data services in the application. This abstract class cannot be instantiated directly and defines the contract that all service implementations must follow. It enforces the implementation of the fetchData() method, ensuring a consistent interface for data retrieval across different data sources.

SheetService Class

Extends the abstract Service class to fetch sensor data from a published Google Spreadsheet using the Visualization API. Parses the raw JSONP response and returns structured data containing temperature and humidity readings from the campus sensor.

SMHIService Class

Extends the abstract Service class to fetch official weather data from the SMHI (Swedish Meteorological and Hydrological Institute) Open Data API. Configurable with a measurement parameter, weather station ID, and time period. Used to retrieve temperature and humidity data for comparison with local sensor readings.

SensorChart Class

A Web Component that subscribes to the Observable and renders a live line chart using ApexCharts, displaying temperature and humidity readings from the campus sensor. The SheetService is instantiated once in connectedCallback and reused across renders. The selected time period is persisted via LocalStorage and restored on load.

SMHIChart Class

A Web Component that fetches and charts official temperature and humidity data from the SMHI Open Data API. Uses a periodMap defined once in connectedCallback to translate UI period labels (Day, Month) into SMHI API period strings. Subscribes to the Observable to respond to period changes.

Observable Class

Implements both the Singleton and Observer design patterns. As a Singleton, only one instance exists across the application. As an Observable, it maintains a list of subscribed chart components and notifies them when the selected time period changes, or triggers a refresh of all charts simultaneously.
